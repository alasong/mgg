## 架构规范

### 分层架构
```
┌─────────────────────────────────────┐
│           展示层 (Presentation)      │
│  - UI组件 / 页面 / 视图              │
│  - 只能依赖接口层                     │
└─────────────────────────────────────┘
                    │
┌─────────────────────────────────────┐
│           接口层 (Interface)         │
│  - API / Controller / Router        │
│  - 只能依赖业务层                     │
└─────────────────────────────────────┘
                    │
┌─────────────────────────────────────┐
│           业务层 (Business)          │
│  - Service / UseCase                │
│  - 只能依赖数据层                     │
└─────────────────────────────────────┘
                    │
┌─────────────────────────────────────┐
│           数据层 (Data)              │
│  - Repository / DAO / Model         │
│  - 只能依赖基础设施                   │
└─────────────────────────────────────┘
                    │
┌─────────────────────────────────────┐
│         基础设施层 (Infrastructure)  │
│  - 数据库 / 缓存 / 消息队列          │
└─────────────────────────────────────┘
```

### 依赖规则
- **单向依赖**：上层可以依赖下层，下层不能依赖上层
- **接口隔离**：层间通过接口通信，不直接依赖实现
- **依赖注入**：使用DI容器管理依赖

## 开发流程

### 分支策略
```
main (生产)
  │
  ├── release/* (发布分支)
  │
  └── develop (开发)
        │
        └── feature/* (功能分支)
```

### 提交规范
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type**: feat, fix, docs, style, refactor, test, chore

**示例**:
```
feat(user): 添加用户注册功能

- 实现用户注册接口
- 添加邮箱验证
- 添加密码强度检查

Closes #123
```

### 发布流程
1. 功能分支开发完成
2. 合并到develop分支
3. 创建release分支
4. 执行完整测试
5. 修复问题
6. 合并到main
7. 打tag发布
