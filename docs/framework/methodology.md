# ADE高效开发规范

## 核心理念

**PDCA循环 × Spec驱动 × Skill执行 × TDD原子实现 × 状态文本固化**

以PDCA为底层运转引擎，每个层级每个阶段都形成闭环。通过AGENT.MD进行项目规范和架构管控，**Spec定义迭代做什么，Skill定义任务怎么做**，分解后继续Spec+Skill协作，原子开发用TDD完成，执行状态通过独立文本文件追踪。

### 核心概念

| 概念 | 定位 | 目录 | 性质 | 示例 |
|------|------|------|------|------|
| **Spec** | 迭代规格，定义"这个阶段做什么" | `docs/specs/` | 项目计划，有PDCA，引用Skill | `iteration-3-spec.md` 引用 data-layer/business-layer/api-layer Skill |
| **Skill** | 开发能力，定义"这个任务怎么做" | `docs/skills/` | 可复用的执行单元，有PDCA、TDD要求 | `data-layer-skill.md` 描述如何实现Repository |
| **执行状态** | 进展记录，定义"做了什么、进展如何" | `docs/execution/` | 动态更新的文本，与Spec一一对应 | `iteration-3-status.md` 记录当前到哪一步 |

### 执行流程

```
1. 加载 Spec → 读取元信息中的"执行状态文件"路径
2. 读取执行状态 → 确定当前进展到哪一步
3. 检查Spec引用的Skill → 如不存在，先在Plan阶段创建
4. 按 PDCA 步骤执行 → 调用引用的 Skill
5. Skill 执行完成 → 更新执行状态文件
6. Spec 文件本身不修改，执行状态文件动态更新
```

### Skill创建规则（强制）

**Spec中引用的所有Skill必须在Plan阶段创建完成，Do阶段开始前所有Skill必须存在。**

| 阶段 | Skill要求 | 禁止行为 |
|------|-----------|----------|
| **Plan** | 检查Spec引用的Skill，不存在的必须创建 | 跳过Skill创建直接进入Do |
| **Do** | 按已有Skill步骤执行TDD | 边做边创建Skill |
| **Check** | 验证Skill执行结果 | 修改Skill定义 |
| **Act** | 优化Skill文档 | 无 |

**执行顺序**：Specs first → Skills → TDD
1. 先有Spec（定义做什么）
2. 再有Skill（定义怎么做）
3. 最后TDD（原子实现）

### 状态刷新规则（强制）

**每次迭代执行后，必须刷新迭代执行状态文件。** 如果有问题，需要循环修正，修正后再次刷新状态。

**刷新时必须检查全局一致性**，不能只改局部：

| 检查位置 | 要求 |
|----------|------|
| 顶部"当前状态"摘要 | 整体进度（Plan/Do/Check/Act）必须与实际一致 |
| 各 PDCA 阶段区块 | 状态（✅/⏳）和执行结果必须与实际一致 |
| 未完成项列表 | 已完成项必须移除，未完成项必须保留 |
| 执行日志 | 最新操作必须追加到日志末尾 |

**循环修正原则**：
- 遇到问题不跳过，循环修正直到通过
- 每次修正后必须再次刷新状态文件
- 状态文件必须如实反映当前进展，不假装已完成

## 二、AGENT.MD 规范管控

### 2.1 文件位置
```
项目根目录/AGENT.MD
```

### 2.2 核心内容结构

```markdown
# [项目名称] AGENT.MD

## 项目概述
- 项目目标
- 核心价值
- 技术栈

## 架构规范
- 分层架构定义
- 模块边界
- 依赖规则
- 通信协议

## 编码规范
- 命名规范
- 代码风格
- 注释规范

## 质量标准
- 测试覆盖率要求
- 代码审查标准
- 性能指标
- 安全要求

## 开发流程
- 分支策略
- 提交规范
- 发布流程

## Skill清单

- Skill依赖关系
```

- **强制性**：所有开发活动必须遵循AGENT.MD定义

- **可演进**：规范随项目演进更新

## 三、Skill驱动开发

### 3.1 Skill定义

Skill是**可复用的能力单元**，封装特定开发活动的：
- 输入条件
- 执行步骤
- 输出产物
- 质量标准

### 3.2 Skill层级

```
层面Skill (Layer)
  ├── 数据层开发Skill
  ├── 业务层开发Skill
  ├── API层开发Skill
  └── 前端开发Skill

功能Skill (Feature)
  ├── ES搜索集成Skill
  ├── 权限管理Skill
  ├── 高级搜索Skill
  ├── 协作功能Skill
  ├── 导入导出Skill
  ├── 数据分析Skill
  ├── 系统优化与部署Skill
  └── 远端增量调测Skill
```

```markdown
- **质量标准**: 验收标准
- **执行状态文件**: `docs/execution/[skill-name]-status.md`

## PDCA执行步骤

### Plan (规划)
- **目标**: ...
- **输入**: ...
- **活动**: ...
- **输出**: ...

### Do (执行)
- **输入**: ...
- **活动**: ...
- **输出**: ...

### Check (验证)
- **输入**: ...
- **活动**: ...
- **输出**: ...

- **输入**: ...
- **活动**: ...
- **输出**: ...
```

### 3.4 Skill与Spec的关系

Spec 引用 Skill，Skill 不引用 Spec：

```
Spec (迭代规格)
  ├── 依赖Skill: data-layer-skill.md
  ├── 依赖Skill: business-layer-skill.md
  └── 依赖Skill: api-layer-skill.md

Skill (开发能力)
  ├── 独立可复用
  ├── 不依赖特定Spec
  └── 可被多个Spec引用
```

### 3.5 Skill开发流程 (PDCA闭环)

```
┌─────────────────────────────────────────────────────┐
│              Skill PDCA 循环                         │
│                                                     │
│  Plan: 识别能力需求 → 定义Skill边界                  │
│   ↓                                                 │
│  Do: 编写Skill文档 → 验证可执行性                    │
│   ↓                                                 │
│  Check: 实际使用 → 收集执行反馈                      │
│   ↓                                                 │
│  Act: 分析反馈 → 优化Skill → 更新版本                │
│   ↓                                                 │
│  └────────────── 回到Plan ──────────────────┘       │
└─────────────────────────────────────────────────────┘
```

**每个阶段的详细活动**:

| PDCA | 活动 | 输入 | 输出 | 验证标准 |
|------|------|------|------|----------|
| **Plan** | 识别能力需求、定义边界 | 任务分解结果 | Skill需求清单 | 需求清晰可执行 |
| **Do** | 编写Skill文档、验证可执行性 | Skill需求清单 | Skill文档v1.0 | 步骤完整无歧义 |
| **Check** | 使用Skill执行、收集反馈 | Skill文档 | 执行日志、问题清单 | 反馈真实具体 |
| **Act** | 分析反馈、优化Skill | 执行日志、问题清单 | Skill文档vN+1 | 问题已解决 |

## 四、迭代分解驱动

### 4.1 分解原则

```
项目 → 阶段 → 层面 → 功能 → 模块 → 组件 → 函数
```

每层分解都遵循：
1. **先定义Skill**：明确该层级的能力需求
2. **再开展活动**：使用Skill执行开发
3. **过程文档化**：记录决策和实现细节
4. **TDD原子实现**：最底层用TDD完成

### 4.2 迭代流程 (PDCA驱动)

**重要原则：先规划，后执行**

```
正确顺序:
1. 生成所有迭代Specs (完整路线图) → 明确全局目标和依赖
2. 每个Spec中引用所需Skill
3. 如Skill不存在 → 创建Skill文档
4. 按迭代顺序执行: 加载Spec → 检查Skill → TDD开发
```

**禁止跳过Spec直接开发**。每个迭代必须先有完整的Spec文档，包含PDCA四个阶段的详细规划，再进入Do阶段。

### 4.3 Spec文档模板 (PDCA完整格式)

**每个Spec必须包含PDCA四个阶段的完整规划**，禁止只写目标和任务分解。

```markdown
# Iteration N Spec - [迭代名称]

## 元信息
- **迭代编号**: N
- **迭代周期**: YYYY-MM-DD ~ YYYY-MM-DD
- **前置迭代**: Iteration N-1
- **关联战略**: [战略文档路径]

## 迭代目标
1. [目标1] - 可衡量的指标
2. [目标2] - 可衡量的指标

## 任务分解
```
Iteration N
├── 模块A
│   ├── 功能A1 → Skill: xxx-skill
│   └── 功能A2 → Skill: yyy-skill
└── 模块B
    └── 功能B1 → Skill: zzz-skill
```

## Skill引用
| Skill | 类型 | 状态 | 说明 |
|-------|------|------|------|
| xxx-skill | 现有/新建 | ✅/❌ | 说明 |

## 验收标准
### 功能验收

- [ ] 功能2：具体描述

### 性能验收
- [ ] 指标1：具体数值
- [ ] 指标2：具体数值

### 质量验收
- [ ] 测试覆盖率 ≥ X%
- [ ] 代码重复率 < Y%
- [ ] 无P0/P1 Bug

## PDCA执行计划

### Plan (规划)
- **目标**: 明确本次迭代要做什么
- **活动**:
  1. 分析竞品差距和战略目标
  2. 确定迭代目标和范围
  3. 分解任务到可执行层级
  4. 识别所需Skill
  5. 制定验收标准
- **输出**: Spec文档、执行状态文件、Skill文档
- **完成标准**: 所有文档完整可执行

### Do (执行)
- **目标**: 按Skill步骤TDD实现
- **活动**:
  1. 加载Spec → 检查引用的Skill
  2. 如Skill不存在 → 创建Skill文档
  3. 按Skill步骤执行开发
  4. 功能级TDD: Red→Green→Refactor
  5. 更新执行状态文件
- **输出**: 代码、测试、执行状态更新
- **完成标准**: 所有功能实现，测试通过

### Check (验证)
- **目标**: 验证功能和质量
- **活动**:
  1. 执行全部测试 (单元+集成+验收)
  2. 代码审查
  3. 质量指标度量 (覆盖率/复杂度/重复率)
  4. 对比验收标准
  5. 性能基准测试
  6. 更新执行状态文件
- **输出**: 测试报告、质量报告
- **完成标准**: 所有验收标准达标

### Act (改进)
- **目标**: 复盘和优化
- **活动**:
  1. 复盘: 做得好的 / 需要改进的
  2. 优化Skill: 根据执行反馈更新Skill文档
  3. 更新规范: 将经验沉淀到AGENT.MD
  4. 固化状态: 更新执行状态文件
- **输出**: 优化后的代码/Skill/规范
- **完成标准**: 经验已沉淀，状态已固化

## 质量指标
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 测试覆盖率 | ≥80% | pytest --cov |
```

## 六、TDD原子开发 (PDCA微循环)

### 6.1 TDD即PDCA

TDD的Red-Green-Refactor本质上就是一个PDCA微循环：

```
┌─────────────────────────────────────────────────────┐
│           TDD = PDCA 微循环                          │
│                                                     │
│  Plan  →  Red: 规划测试用例，定义预期行为             │
│   ↓        "我要实现什么，怎么验证"                   │
│                                                     │
│  Do    →  Green: 编写最小实现让测试通过               │
│   ↓        "用最少的代码满足测试"                     │
│                                                     │
│  Check →  运行测试验证                                │
│   ↓        "测试是否通过，行为是否正确"               │
│                                                     │
│  Act   →  Refactor: 重构优化，消除坏味道              │
│            "代码是否清晰，是否可以更好"               │
│            ↓                                        │
│            └──── 下一个循环 ────┘                    │
└─────────────────────────────────────────────────────┘
```

### 6.2 TDD-PDCA映射详解

| PDCA | TDD阶段 | 核心问题 | 活动 | 产出 |
|------|---------|----------|------|------|
| **Plan** | Red | 这个功能应该做什么？怎么验证？ | 编写测试用例，定义输入输出和预期行为 | 失败的测试 |
| **Do** | Green | 最少需要多少代码让测试通过？ | 编写最小实现，不追求完美 | 通过的测试 |
| **Check** | 测试运行 | 实现是否正确？有无遗漏？ | 运行全部测试，检查覆盖率 | 测试报告 |
| **Act** | Refactor | 代码质量如何？能否更好？ | 消除重复、改善命名、提取方法 | 更清晰的代码 |

### 6.3 TDD流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│   Red   │───▶│  Green  │───▶│Refactor │
│ 写失败  │    │ 最小实现│    │ 重构优化│
│  测试   │    │  代码   │    │  代码   │
└─────────┘    └─────────┘    └─────────┘
      │                              │
      └──────────────────────────────┘
                  循环直到功能完成
```

### 6.4 TDD规则

1. **先写测试**：在编写任何功能代码前，先写测试
2. **测试失败**：确保测试失败（Red）
3. **最小实现**：只写让测试通过的最少代码（Green）
4. **重构优化**：在测试保护下重构代码（Refactor）
5. **持续循环**：重复直到功能完整

### 6.5 TDD与Skill结合

```
Skill定义功能边界
    │
    ▼
TDD实现原子功能
    │
    ├── Red: 根据Skill定义写测试
    ├── Green: 实现Skill要求的逻辑
    └── Refactor: 优化代码结构
    │
    ▼
测试通过 → 记录TDD日志 → 继续下一个循环
```

### 6.6 TDD示例

```python
# 1. Red - 写失败测试
def test_user_registration():
    service = UserService()
    result = service.register("test@example.com", "password123")
    assert result.success == True
    assert result.user.email == "test@example.com"

# 2. Green - 最小实现
class UserService:
    def register(self, email, password):
        return RegistrationResult(success=True, user=User(email=email))

# 3. Refactor - 重构优化
class UserService:
    def __init__(self, user_repository=None):
        self.user_repository = user_repository or UserRepository()
    
    def register(self, email, password):
        self._validate_email(email)
        self._validate_password(password)
        user = User(email=email, password_hash=self._hash_password(password))
        self.user_repository.save(user)
        return RegistrationResult(success=True, user=user)
```

## 九、最佳实践

### 9.1 PDCA实践
- **Plan要具体**：目标可量化，验收标准明确
- **Do要记录**：边做边记，不事后补
- **Check要客观**：用数据说话，不凭感觉
- **Act要落地**：改进措施具体可执行，纳入下一轮Plan

### 9.2 Skill开发
- **小步快跑**：Skill粒度要小，易于执行和验证
- **持续优化**：每次Act阶段根据执行反馈优化Skill
- **版本管理**：Skill要有版本号，便于追溯

### 9.3 迭代分解
- **垂直切分**：按功能垂直切分，而非水平切分
- **可交付**：每个迭代都要有可交付的产物
- **风险前置**：高风险任务优先处理
- **PDCA闭环**：每个迭代必须完成完整的PDCA循环

### 9.4 TDD实施
- **测试先行**：严格遵循Red-Green-Refactor (Plan-Do-Act)
- **测试隔离**：每个测试独立，不依赖执行顺序
- **测试命名**：测试名称要清晰表达意图
- **微循环不跳过**：每个PDCA微循环都要完整执行

### 9.5 文档固化
- **实时记录**：边做边记，不要事后补
- **结构化**：使用模板，保持文档结构一致
- **可追溯**：文档要能追溯到具体决策和实现
- **Act阶段归档**：每个迭代结束时统一归档

### C. 端口管理规范

#### C.1 端口管理文件
每个部署环境必须维护 `docs/ports.toml` 文件，记录所有服务的端口分配、映射关系和冲突情况。

#### C.2 端口管理原则
1. **唯一性**：同一端口不能被多个服务占用
2. **内外分离**：外部访问端口与内部容器端口分开管理
3. **冲突记录**：所有端口冲突必须记录在 `conflicts` 段中
4. **部署前检查**：部署前必须检查端口占用情况，避免冲突

#### C.3 端口管理流程
```
1. 部署前执行 ss -tlnp 检查端口占用
2. 检查已有容器和进程的端口映射
3. 更新 ports.toml 文件
4. 如有冲突，记录到 conflicts 段并制定解决方案
5. 部署后验证端口映射是否正确
```

#### C.4 ports.toml 结构
```toml
[server]                    # 服务器基本信息
host = "IP地址"
network = "Docker网络名称"

[external_ports]            # 外部访问端口（宿主机端口映射）
http = 80
https = 443

[internal_ports]            # 内部服务端口（容器网络内部）
service_name = { container = "容器名", port = 端口, protocol = "tcp", description = "描述" }

[access_urls]               # 外部访问地址
frontend = "http://IP"
api = "http://IP/api/v1"

[internal_urls]             # 容器网络内部地址
backend = "http://容器名:端口"

[conflicts]                 # 端口冲突记录
conflict_name = { status = "resolved|active", description = "描述" }
```

#### C.5 常见端口冲突处理
| 冲突类型 | 处理方式 |
|----------|----------|
| 宿主机nginx占用80 | 停止宿主机nginx，使用容器nginx |
| iptables REDIRECT规则 | 删除REDIRECT规则 |
| 其他服务占用端口 | 更换服务端口或停止冲突服务 |
| 容器端口冲突 | 修改容器端口映射 |
