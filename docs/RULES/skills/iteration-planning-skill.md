# 迭代规划Skill

## 元信息
- **类型**: 流程
- **前置条件**: 竞品分析报告、战略文档、当前执行状态
- **输出产物**: Spec文档、执行状态文件、Skill引用清单
- **质量标准**: 目标清晰可衡量、任务分解完整、验收标准明确
- **执行状态文件**: `docs/execution/iteration-planning-skill-status.md`

## 简介

迭代规划Skill封装了基于PDCA规范的迭代规划完整流程，用于：
1. 分析竞品差距和战略目标
2. 确定迭代目标和范围
3. 分解任务到可执行层级
4. 编写Spec文档和执行状态文件
5. 制定验收标准和质量指标

## PDCA执行步骤

### Plan (规划)

**目标**: 明确本次迭代要做什么，做到什么程度

**输入**:
- 竞品对比分析报告 (`docs/comparison-*.md`)
- 战略对齐文档 (`docs/strategy-*.md`)
- 当前执行状态 (`docs/execution/*-status.md`)
- 现有Skill清单 (`docs/skills/`)

**活动**:
1. 分析竞品差距，识别优先级最高的能力缺口
2. 确定本次迭代目标（聚焦1-2个核心能力）
3. 分解任务到可执行层级（模块→功能→原子）
4. 识别所需Skill（新建或引用现有）
5. 制定验收标准（功能+性能+质量）
6. 评估工作量和风险

**输出**:
- 迭代目标清单
- 任务分解树
- Skill引用清单
- 验收标准
- 风险评估

**验证标准**:
- 目标清晰可衡量（有具体指标）
- 任务分解到原子级别（可TDD实现）
- 验收标准可验证（有测试方法）
- 工作量合理（1-2周可完成）

### Do (执行)

**目标**: 编写Spec文档和执行状态文件

**输入**:
- Plan阶段输出

**活动**:
1. 创建Spec文档 (`docs/specs/iteration-N-spec.md`)
   - 迭代目标
   - 任务分解
   - Skill引用
   - 验收标准
   - 质量指标
2. 创建执行状态文件 (`docs/execution/iteration-N-status.md`)
   - 关联Spec
   - 当前状态
   - PDCA执行记录模板
   - TDD执行摘要模板
3. 如需新Skill，创建Skill文档 (`docs/skills/xxx-skill.md`)

**输出**:
- Spec文档
- 执行状态文件
- 新Skill文档（如有）

**验证标准**:
- Spec文档结构完整
- 执行状态文件可追踪
- Skill文档可执行

### Check (验证)

**目标**: 验证Spec的完整性和可行性

**输入**:
- Spec文档
- 执行状态文件
- Skill文档

**活动**:
1. Spec文档审查
   - 目标是否清晰可衡量
   - 任务分解是否完整
   - 验收标准是否可验证
2. 技术可行性评估
   - 技术方案是否可行
   - 依赖是否满足
   - 风险是否可控
3. 工作量评估
   - 是否可在1-2周内完成
   - 是否需要额外资源
4. 质量保障评估
   - TDD是否可执行
   - 测试覆盖率目标是否合理
   - 性能基准是否可测量

**输出**:
- 审查意见
- 修改建议

**验证标准**:
- 无P0级别问题
- 工作量合理
- 质量保障可行

### Act (改进)

**目标**: 优化Spec，沉淀规划经验

**输入**:
- 审查意见
- 修改建议

**活动**:
1. 根据反馈修改Spec
2. 优化任务分解
3. 调整验收标准
4. 沉淀规划经验到规范文档
5. 更新执行状态文件

**输出**:
- 优化后的Spec文档
- 更新后的执行状态文件
- 规划经验总结

**验证标准**:
- 所有P0问题已解决
- Spec文档最终版本确认
- 执行状态文件可开始执行

## 迭代规划模板

```markdown
# Iteration N Spec - [迭代名称]

## 元信息
- **迭代编号**: N
- **迭代周期**: YYYY-MM-DD ~ YYYY-MM-DD
- **前置迭代**: Iteration N-1
- **关联战略**: [战略文档路径]

## 迭代目标
1. [目标1] - 可衡量的指标
2. [目标2] - 可衡量的指标

## 任务分解
```
Iteration N
├── 模块A
│   ├── 功能A1 → Skill: xxx-skill
│   └── 功能A2 → Skill: yyy-skill
├── 模块B
│   └── 功能B1 → Skill: zzz-skill
└── 模块C
    └── 功能C1 → 新建Skill
```

## Skill引用
| Skill | 类型 | 状态 | 说明 |
|-------|------|------|------|
| xxx-skill | 现有 | ✅ | 直接使用 |
| yyy-skill | 现有 | ⚠️ | 需增强 |
| zzz-skill | 新建 | ❌ | 需创建 |

## 验收标准
### 功能验收
- [ ] 功能1：具体描述
- [ ] 功能2：具体描述

### 性能验收
- [ ] 指标1：具体数值
- [ ] 指标2：具体数值

### 质量验收
- [ ] 测试覆盖率 ≥ X%
- [ ] 代码重复率 < Y%
- [ ] 无P0/P1 Bug

## 质量指标
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 测试覆盖率 | ≥80% | pytest --cov |
| 代码重复率 | <5% | radon cc |
| API响应时间 | <X ms | 基准测试 |

## 风险评估
| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| ... | ... | ... | ... |

## 执行状态
- **当前阶段**: Plan ⏳ → Do ⏳ → Check ⏳ → Act ⏳
- **最后更新**: YYYY-MM-DD
```

## 依赖关系
- 依赖：竞品分析流程、战略对齐流程
- 被依赖：所有迭代Spec

## 注意事项
1. **聚焦原则**: 每迭代聚焦1-2个核心能力，不贪多
2. **可验证**: 所有验收标准必须有可执行的验证方法
3. **TDD保障**: 每个原子功能必须可TDD实现
4. **渐进式**: 基础→核心→应用→超越，不跳跃
5. **质量优先**: 不牺牲质量换速度
